GrantSearch NZ — How It Works

End-to-end overview of the grant search, deep search, and application tracking pipeline

DB
Grants Database (Neon Postgres)

The grants database is the foundation of the search pipeline. It contains ~4,500 active grants from NZ funders, populated via a multi-step data pipeline and continuously enriched. Grants are pre-tagged with regions and sectors so searches can filter and rank without any external API calls.

charities table
~6,300 records from the NZ Charities Register + 272 curated funders. Each row has a curated_grant_url and regions[]. Source of truth for funder identity.
grants table
Individual grant programs linked to funders via funder_id. Fields: name, type, description, url, amount range, deadline, regions[], sectors[], eligibility[], application_form_url. GIN indexes on regions & sectors for fast overlap queries.
Runtime storage
4 tables created on first use: saved_searches, shortlisted_grants, deep_searches, grant_applications. All store JSON payloads alongside metadata.

1
Grant Search (Two-Pass Pipeline)

User submits a search form with their org details, sectors, regions, and funding purpose. The API queries the grants DB and runs a two-pass AI pipeline: cheap triage then full scoring. No web scraping or external search — only GPT-4o-mini token costs.

Search pipeline stages
Stage Output Description Cost
DB query ~500–4,500 SQL query filters by region overlap & orders by sector overlap count. Returns all active matching grants. free
Pass 1: Triage ~200–400 GPT-4o-mini classifies each grant as RELEVANT or SKIP in batches of 100. Binary decision — cheap & fast. Skips only when ≥90% confident grant is irrelevant. GPT-4o-mini
Pass 2: Scoring ~200–400 GPT-4o-mini scores each relevant grant on 3 dimensions (alignment, ease, attainability) in batches of 25. Generates orgSummary on first batch. GPT-4o-mini
Alignment filter ~40–80 Grants with alignment < 5.0 removed from final results free

2
Scoring System

Every grant is scored on three dimensions by GPT-4o-mini. The model evaluates mission fit, application complexity, and win probability based on the org context and grant description.

Alignment (50%)
How well the grant purpose and form of support match the org's mission and funding request.
0–3 poor | 4–6 partial | 7–8 good | 9–10 designed for this
Attainability (30%)
Likelihood this org wins given competition, eligibility fit, and regional relevance.
1–2 very competitive | 5–6 moderate | 9–10 few applicants
Ease (20%)
Application complexity — higher is simpler.
1–2 multi-stage | 5–6 full proposal | 9–10 simple online form
Overall = (alignment × 0.5) + (attainability × 0.3) + (ease × 0.2), rounded to 1dp

3
Deep Search (Per-Grant Research)

Triggered manually per grant. Uses Serper + Tavily + GPT-4o to find precise details that the initial database may not have: exact amounts, deadlines, application forms, eligibility criteria, and required documents. Recalibrates scores based on findings. Discovered data is written back to the grants DB so future searches benefit.

Deep search phases
Phase Description Service
1. Web search 6–8 parallel Serper queries targeting application forms, guidelines, deadlines, past recipients for this specific grant Serper
2. Content extraction Tavily extracts full page content from up to 10 discovered URLs, prioritising the funder's own domain Tavily
3. AI analysis GPT-4o synthesises all content into structured output: amounts, dates, eligibility, checklist, contacts, score recalibration with reasoning GPT-4o
4. DB write-back Discovered fields (application_form_url, amount_min, amount_max, deadline) written back to grants table — only fills NULLs, never overwrites free

4
User Workflow

After search results appear, users can shortlist promising grants, run deep searches for more detail, and track applications through to outcome.

/results Search Results
Filter by score, sort, group by funder. Shortlist grants or trigger deep search. Auto-saved for future reference.
/shortlisted Shortlist
Grouped by search. View deep-search insights, eligibility, checklists. Start application from here.
/applications Applications
Track status: Preparing → Submitted → Under Review → Approved/Declined. Notes, financial tracking, timeline.
⚙
Data Pipeline (Offline Scripts)

One-time and periodic scripts that populate and enrich the grants database. Run locally via DATABASE_URL="..." npx tsx scripts/<script>.ts.

# Script Description Status
1 import-charities.ts Import ~6,310 charities from NZ Charities Register API. Batched inserts, safe to re-run. done
2 import-curated-funders.ts Import 272 hand-curated funders from nz.ts with grant URLs and regions. Cross-source dedup with register. done
3 enrich-charities.ts Scrape funder websites, extract grant programs via GPT-4o-mini, insert into grants table. done
4 classify-grant-makers.ts GPT-4o classifies which charities are actual grant-makers. 1,762 non-grant-maker grants deactivated. done
5 fill-missing-fields.ts Targeted re-enrichment for amounts, deadlines, and form URLs on grants with missing data. done
6 retag-new-sectors.ts Re-tag all grants with new sector IDs (economic-development, animal-welfare) via GPT-4o-mini. done
7 db-quality-fixes.ts Merge errors, dedup funders, fix homepage URLs, deactivate contamination grants. done
8 fill-apply-urls.ts Find application form URLs via Tavily extract + search. Groups by source_url. Dry-run by default. available
9 fill-missing-fields-tavily.ts Re-enrichment using Tavily (bypasses 403s / JS-rendered pages) for eligibility, amounts, deadlines. available
