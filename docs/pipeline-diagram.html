GrantSearch NZ — How It Works

End-to-end overview of the multi-tenant grant search, deep research, document management, and application tracking platform

DB
Database (Neon Postgres) — ~10 Tables

The database is the foundation of the platform. It contains ~4,500 active grants from NZ funders, populated via a 24-script data pipeline and continuously enriched. All runtime tables support multi-tenancy via organisations — composite keys with org_id ensure data isolation between teams.

Grant Data (2 tables)
charities — ~6,300 records from the NZ Charities Register + 272 curated funders. Each row has curated_grant_url and regions[]. Source of truth for funder identity.

grants — Individual grant programs linked to funders via funder_id. Fields: name, type, description, url, amount range, deadline, regions[], sectors[], eligibility[]. GIN indexes on regions & sectors for fast overlap queries.
Auth & Tenancy (3 tables)
organisations — Tenant boundary. Name, slug. All runtime data scoped to an org.

users — Authenticated users linked to an org. Email, name, role (admin/member), managed via NextAuth v5.

invitations — Token-based team invites. Email, role, expiry, accepted status.
Runtime Storage (5 tables)
All tenant-scoped with org_id composite keys:

saved_searches — Search parameters and result snapshots
shortlisted_grants — Grants saved for further review
deep_searches — Per-grant research results from deep search
grant_applications — Application tracking with status workflow
documents — Document metadata for Vercel Blob uploads, with categories and usage tracking
Checklist System (2 tables)
application_checklist_items — Initialized from deep search findings. Tracks required documents, forms, and steps for each application. Status: pending/complete.

checklist_documents — Links uploaded documents to specific checklist items. Many-to-many join table.

1
Grant Search (Two-Pass Pipeline)

User completes a 4-step search wizard: Organisation → Location/Focus → Funding Details → Review. The API queries the grants DB and runs a two-pass AI pipeline: cheap triage then full scoring. Results are delivered as streaming SSE responses so the UI updates progressively. No web scraping or external search — only GPT-4.1-mini token costs.

Search pipeline stages
Stage Output Description Cost
DB query ~500–4,500 SQL query filters by region overlap & orders by sector overlap count. Returns all active matching grants. free
Pass 1: Triage ~200–400 GPT-4.1-mini classifies each grant as RELEVANT or SKIP in batches of 100. Binary decision — cheap & fast. Skips only when ≥90% confident grant is irrelevant. GPT-4.1-mini
Pass 2: Scoring ~200–400 GPT-4.1-mini scores each relevant grant on 3 dimensions (alignment, ease, attainability) in batches of 25. Generates orgSummary on first batch. Results streamed via SSE. GPT-4.1-mini
Alignment filter ~40–80 Grants with alignment < 5.0 removed from final results free

2
Scoring System

Every grant is scored on three dimensions by GPT-4.1-mini. The model evaluates mission fit, application complexity, and win probability based on the org context and grant description.

Alignment (50%)
How well the grant purpose and form of support match the org's mission and funding request.
0–3 poor | 4–6 partial | 7–8 good | 9–10 designed for this
Attainability (30%)
Likelihood this org wins given competition, eligibility fit, and regional relevance.
1–2 very competitive | 5–6 moderate | 9–10 few applicants
Ease (20%)
Application complexity — higher is simpler.
1–2 multi-stage | 5–6 full proposal | 9–10 simple online form
Overall = (alignment × 0.5) + (attainability × 0.3) + (ease × 0.2), rounded to 1dp

3
Deep Search (Per-Grant Research)

Triggered manually per grant. Uses Serper + Tavily + GPT-4o to find precise details that the initial database may not have: exact amounts, deadlines, application forms, eligibility criteria, and required documents. Recalibrates scores based on findings. Discovered data is written back to the grants DB so future searches benefit. Also extracts application checklist items which are saved to the DB for tracking.

Deep search phases
Phase Description Service
1. Web search 6–8 parallel Serper queries targeting application forms, guidelines, deadlines, past recipients for this specific grant Serper
2. Content extraction Tavily extracts full page content from up to 10 discovered URLs, prioritising the funder's own domain Tavily
3. AI analysis GPT-4o synthesises all content into structured output: amounts, dates, eligibility, checklist items, contacts, score recalibration with reasoning GPT-4o
4. DB write-back Discovered fields (application_form_url, amount_min, amount_max, deadline) written back to grants table — only fills NULLs, never overwrites. Checklist items saved to application_checklist_items. free

4
User Workflow & Pages

Multi-tenant platform with authentication, team management, document storage, and full application tracking. Users sign up, create or join an organisation, then search, shortlist, research, and manage grant applications. Dark mode supported via theme toggle (light/dark/system).

/search Search Wizard
4-step wizard: Organisation → Location/Focus → Funding Details → Review. Streams results via SSE as scoring completes. Auto-saves search for future reference.
/results Search Results
Filter by score, sort, group by funder. Shortlist grants or trigger deep search. View scoring breakdown per grant.
/shortlisted Shortlist
Grouped by search. View deep-search insights, eligibility, checklists. Start application from here.
/applications Applications
Track status: Preparing → Submitted → Under Review → Approved/Declined. Checklist system initialized from deep search — track required items and attach documents. Notes, financial tracking, timeline.
/documents Document Library
Upload documents to Vercel Blob storage. Categorize by type (financial, governance, project plans, etc.). Usage tracking shows which applications reference each document.
/settings Settings
Organisation settings and profile management. Team member list, role management (admin/member). Invite new members via email with token-based links.
/login /signup /invite/[token] Auth
NextAuth v5 authentication with credentials provider. Sign up creates a new organisation. Invitation tokens allow joining an existing organisation. Session-based with JWT.

Data Pipeline (24 Offline Scripts)

One-time and periodic scripts that populate, enrich, and maintain the grants database. Run locally via DATABASE_URL="..." npx tsx scripts/<script>.ts. Grouped by function below.

Import & Discovery
import-charities.ts Import ~6,310 charities from NZ Charities Register API. Batched inserts, safe to re-run.
import-curated-funders.ts Import 272 hand-curated funders from nz.ts with grant URLs and regions. Cross-source dedup.
discover-websites.ts Find funder websites via web search for charities without URLs.
discover-from-directories.ts Discover new funders and grants from NZ funding directories.
discover-trustee-grants.ts Find grants from trustee companies and community trusts.
Enrichment
enrich-with-tavily.ts Enrich grant data using Tavily content extraction — bypasses 403s and JS-rendered pages.
enrich-with-playwright.ts Enrich grants using headless browser (Playwright) for sites that block API scraping.
fill-missing-fields.ts Targeted re-enrichment for amounts, deadlines, and form URLs on grants with missing data.
fill-missing-fields-tavily.ts Tavily-based re-enrichment for eligibility, amounts, deadlines where direct scraping fails.
fill-apply-urls.ts Find application form URLs via Tavily extract + search. Groups by source_url. Dry-run by default.
fill-grant-regions.ts Populate missing region tags on grants based on funder data and content analysis.
batch-deep-search.ts Run deep search in bulk across multiple grants to backfill detailed data.
Classification
classify-grant-makers.ts GPT classifies which charities are actual grant-makers vs. service providers.
classify-funder-types.ts Classify funders by type (community trust, government, corporate, etc.) using rule-based logic.
classify-funder-types-gpt.ts GPT-based funder type classification for cases where rules are insufficient.
find-individual-grants.ts Identify and tag grants aimed at individuals (scholarships, personal grants) vs. organisations.
Quality & Maintenance
dedup-grants.ts Find and merge duplicate grant entries across funders.
deactivate-expired.ts Deactivate grants with past deadlines to keep search results current.
fix-dead-urls.ts Detect and fix broken URLs across grants and funder records.
verify-grant-urls.ts Bulk-verify that grant and application URLs are still live and returning valid content.
check-orphaned-ids.ts Find grants referencing non-existent funder IDs or other data integrity issues.
clear-user-data.ts Wipe runtime user data (searches, shortlists, applications) for dev/testing resets.
Reporting & Analysis
analyse-scoring.ts Analyse scoring distribution and calibration across search results.
coverage-report.ts Report on data completeness: how many grants have amounts, deadlines, URLs, etc.
