Discovery & Enumeration

Three parallel workstreams: discover funder URLs from directories, enumerate all known funders via GPT, and look up local charities DB.

Runs in parallel
Tavily Step 0 — Directory Discovery
Extract org website for context (1 URL).
Extract grant directory pages to find funder URLs.
Discover additional directories via search.
~12 grant directories × Tavily extract
GPT-4o Step 1 — Funder Enumeration
4 parallel GPT-4o calls, each with a subset of funder type categories (gaming trusts, govt, corporate, etc.).
1 GPT-4o call for programme enumeration.
Each returns named funders with search queries and region tags.
~19 funder type hints → 4 category groups → no fixed limit (typically ~170 funders)
DB Charities Register Lookup
SQLite full-text search on NZ Charities Register.
Matches by sector + region + funding purpose.
Enriched funders (with grant summary) bypass Tavily entirely — injected as synthetic pages at Step 3d.
GPT-4o-mini Regional Query Generation
Generates 20–30 search queries per search targeting small/obscure local funders: community trusts, district councils, family foundations, sub-regional areas.
↓ Region filter — Enumerated funders filtered to user's selected regions + national. Out-of-region councils/trusts removed before any Serper queries run.
Output → Enumerated funders (region-filtered) + programmes • Discovered URLs • DB funders • Regional queries • Org content

Multi-Source Web Search

9 search strategies run in parallel, each Serper query returns ~10 results. All hits are pooled into rawSearchHits with origin tags for diagnostics.

Concurrent — 20 parallel connections
Serper Search Sources → rawSearchHits[]
  • Enum Searches One Serper query per enumerated funder/programme — searches for their grant application page
  • Seed Queries Static market-wide queries from buildSeedQueries() e.g. "NZ community grants apply 2026"
  • Regional 20-30 GPT-generated queries targeting local/obscure funders in user's regions
  • Directory Deep-dive site:domain searches within known grant directories for user's sector/purpose
  • Purpose Seeds GPT-4o-mini generates purpose-specific queries from org's funding need
  • Associative GPT-4o gap analysis: finds funders that should exist but weren't enumerated
  • Broad Category Automated combinations of funder terminology × sectors × regions
Output → rawSearchHits[] — thousands of URL + snippet pairs, each tagged with discovery origin
2b
Gap-Fill Search

GPT-4o-mini analyses which funder categories are missing from the domains found so far, then generates targeted queries to fill the gaps.

GPT-4o-mini Serper Gap Analysis + Search
Input: list of all domains found so far.
GPT identifies missing funder types, generates targeted queries.
Serper runs each query → results added with origin: 'gap-fill'.
2c
Per-Domain Site Crawl

Many funder searches return homepages. Crawl within each discovered domain to find grant/apply pages buried deeper in the site.

Serper Domain Crawl
For each domain found in rawSearchHits:
4 site:domain queries × ["grants apply", "funding apply", "community grants", "grant application"]
Results added with origin: 'site-crawl'.
Crawl Filters (cost control)
1. Skip domains already at per-domain URL cap (3+ URLs found) — crawling would discover URLs that get capped anyway.

2. Cap total crawl domains at 200 — tail domains produce diminishing returns at 4 queries each.
Without caps: 652 domains × 4 queries = 2,608 Serper queries
Output → rawSearchHits[] now complete — all search strategies have contributed
▼
URL Deduplication, Merge & Filtering

All URL sources are merged, deduplicated, filtered for grant-relevance, and capped per-domain before extraction.

1 Deduplicate rawSearchHits by normalised URL — first-seen wins (preserves origin tag)
2 Merge three URL sources: curated market URLs (region-filtered) + Step 0 discovered URLs + deduplicated search hits. Priority order: curated > directory > search.
3 Grant page filter — isGrantPage(url) removes non-grant URLs (careers, login, news, about pages). Curated URLs bypass this filter.
4 Per-domain cap — max 3 URLs per domain to avoid over-extracting one funder. Excess URLs silently discarded.
Output → allUrlsToExtract[] — final list of URLs to send to content extraction
3
Content Extraction

Extract page content from all URLs via Tavily, with multiple fallback layers for failed extractions. This is typically the most expensive step (~91% of total cost).

Tavily 3a — Primary Extraction
Tavily extracts full page content from every URL in allUrlsToExtract.
Concurrent batch processing.
$0.008 per URL — dominant cost driver
Serper 3b — Search Fallback
URLs that Tavily couldn't extract (JS-heavy sites, bot blocking, timeouts) — retry via site:domain Serper search to get snippet content.
3c — Snippet Promotion
Still-unextracted URLs get their original Serper search snippet promoted as content (if >200 chars). Last resort before giving up.
DB 3d — DB Enrichment Injection
Enriched Charities Register funders (those with grantSummary) injected as synthetic pages. Zero Tavily cost — content comes from the database.
Output → extractedPages[] — { url, content } pairs ready for grant extraction
4
Grant Extraction

GPT reads extracted page content and identifies individual grant programmes with structured metadata.

↓ Pre-filter — isGrantPage(url) applied again to extracted pages. Non-grant pages removed before GPT processing.
GPT-4o-mini Page Batch Processing
Pages batched in groups of 4.
Each batch sent to GPT-4o-mini with the extraction prompt.
GPT extracts: name, funder, type, description, amount range, URL for every grant found on those pages.
15 concurrent batch calls.
e.g. 1,200 pages → 300 batches → 300 GPT calls
↓ Deduplicate grants by normalised name+funder — merges duplicates found across multiple pages
Output → allDiscovered[] — unique DiscoveredGrant objects with provenance tracking via grantOriginMap
5
Scoring & Ranking

GPT-4o scores every discovered grant against the organisation's profile, purpose, region, and funding amount.

GPT-4o Batch Scoring
Grants batched in groups of 25.
Each batch scored by GPT-4o with full org context + page content (2000 char cap per grant).
15 concurrent scoring calls.
Alignment
50%
Attainability
30%
Ease of Apply
20%
Scoring checks:
• Regional relevance (org regions vs grant regions)
• Form-of-support (cash vs in-kind vs services)
• Amount alignment (grant range vs amount sought)
• Sector/purpose match
Score Formula
overall = (alignment × 0.5)
         + (attainability × 0.3)
         + (ease × 0.2)
Each dimension scored 0–10.
First batch also generates orgSummary.
↓ Alignment threshold — grants with alignment < 5 are removed from final results
Output → Scored GrantOpportunity[] sorted by overall score, with alignmentReason, applicationNotes, attainabilityNotes
