1. Pipeline Overview & Architecture

The pipeline turns raw data about New Zealand charities into a searchable database of grants that those charities offer. It runs as a series of TypeScript scripts executed sequentially from the command line. Each step reads from and writes to a shared PostgreSQL database (hosted on Neon).

High-level flow

IMPORT charities         →  FIND their websites     →  FIND their grant pages
  (register + curated)        (HEAD check + validate)       (nav links + search)

→  CLASSIFY as grant-maker  →  EXTRACT grants via GPT  →  DEDUPLICATE & CLEAN
     (rules + content + GPT)      (page content → JSON)       (similarity + GPT audit)

The pipeline has 21 scripts grouped into 8 phases. Each script is idempotent — it can be re-run safely. An ON CONFLICT DO UPDATE pattern on database inserts means re-running a step updates existing records rather than creating duplicates.

Key architectural principle (V4)

Prevention over cure. V3 caught problems in post-hoc deduplication (Step 06a). V4 adds three new mechanisms to prevent problems at the source:

  1. Linked-funder guard — The GPT extraction prompt now explicitly tells GPT to exclude grants belonging to other organisations mentioned on the page, and requires GPT to name which funder each grant belongs to.
  2. URL-level dedup before extraction — When multiple charities share the same grant page URL, only one is sent to GPT. The others are skipped, preventing wasted API calls and cross-funder contamination.
  3. Post-extraction ownership validation — After GPT extraction, a similarity check verifies each extracted grant's stated funder matches the charity being processed. Mismatches are filtered out.

V4 FIX: Two additional OData queries are added to the import:
  • PurposeToGiveGrantsAndDonations eq true — catches any charity that self-declares it gives grants, regardless of its name
  • OrganisationGeneratesFundsGrantsDonationsToOthers eq true — catches charities that generate funds for redistribution to others
This closes the gap where a charity named (for example) "Rata" with no matching name keywords would be missed even though the register flags it as a grant-giver. These queries cost nothing (free API) and any false positives are filtered out by the classification steps downstream (Steps 04a–04c).
Remaining assumption: A charity that gives grants but has none of the 9 name keywords, does NOT have MainActivityId=3, AND has both PurposeToGiveGrantsAndDonations and OrganisationGeneratesFundsGrantsDonationsToOthers set to false (or null) will still be missed. This would require the charity to have incorrectly filled in all relevant register fields — unlikely but possible.

4. Phase 1 — Import & Discovery

Goal: build the universe of potential grant-giving organisations

Step 00

Database Migration

File: scripts/pipeline-v3/00-migrate.ts   Free

Creates all required database tables and columns if they don't exist. Uses CREATE TABLE IF NOT EXISTS and ALTER TABLE ADD COLUMN IF NOT EXISTS so it is safe to re-run.

What it creates

  • charities table — one row per known funder, with columns for classification, website validation, grant page resolution, and enrichment status
  • grants table — one row per extracted grant programme, linked to charities via funder_id
  • pipeline_runs table — logs each pipeline step's execution time, input/output counts, and summary stats
  • Indexes for common queries (funder lookups, active grants, grant page discovery)
Step 01a

Import from NZ Charities Register

File: scripts/pipeline-v3/01a-import-register.ts   OData API Free

What happens

  1. Runs 11 separate OData queries against odata.charities.govt.nz, each fetching charities matching one filter:
    • MainActivityId = 3 (charities whose primary activity is "making grants/loans")
    • Name contains "Trust", "Foundation", "Fund", "Endowment", "Charitable", "Grant", "Gaming", or "Community"
    • V4 NEW: PurposeToGiveGrantsAndDonations = true (charity self-declares it gives grants — catches grant-makers with unusual names that don't match any keyword)
    • V4 NEW: OrganisationGeneratesFundsGrantsDonationsToOthers = true (charity generates funds for redistribution to others)
    Each query also requires RegistrationStatus = 'Registered'.
  2. Results are fetched in pages of 1,000 records until exhausted.
  3. Deduplication: Results from all 11 queries are merged by CharityRegistrationNumber. If the same charity matches multiple queries (e.g., name contains both "Trust" and "Community", or matches both a name keyword and PurposeToGiveGrantsAndDonations), it is kept only once.
  4. Deregistered filter: Charities with a DeregistrationDate or RegistrationStatus != 'Registered' are removed.
  5. Soft exclusion: Names matching a regex pattern for schools, churches, sports clubs, Rotary/Lions, scouts, etc. are flagged with excluded_pattern = true but still imported. They are not deleted — they may still be grant-makers.
  6. Database upsert: Inserted in batches of 500. Uses ON CONFLICT (charity_number) DO UPDATE so re-running refreshes names, purposes, and flags without duplicating rows.

Output

~10,000–12,000 charity rows with source = 'register'.

Decision: Import filter strategy — Charities are imported if they match ANY of: 8 name-keyword patterns, MainActivityId=3, PurposeToGiveGrantsAndDonations=true, or OrganisationGeneratesFundsGrantsDonationsToOthers=true. This avoids importing ~16,000 charities that are very unlikely to be grant-makers (reducing noise). The three non-name filters (MainActivityId, PurposeToGiveGrants, GeneratesFunds) catch grant-makers with unusual names, but only if the charity correctly filled in the relevant register fields.
Decision: Soft exclusion for schools/churches/sports — These are flagged, not deleted. The flag is used later in classification (Step 04a Rule 6) to mark them as "not grant-maker" unless they have a positive signal (like PurposeToGiveGrantsAndDonations = true).
Assumption: The OData API is complete and accurate. If the Charities Register has data quality issues (wrong activity codes, outdated websites), those errors propagate into our database.
Step 01b

Import Curated Funders

File: scripts/pipeline-v3/01b-import-curated.ts   Free

What happens

  1. Reads the 251 curated funder URLs from lib/markets/nz.ts.
  2. Within-list dedup: If the same normalised URL appears twice, the second is skipped.
  3. Cross-source dedup: For each curated URL, checks if a register charity already exists with a matching website hostname.
    • If a match is found: the curated URL is added to the existing register record as curated_grant_url. No new row is created.
    • If no match: a new row is created with source = 'curated' and a synthetic charity number (CU + 8 hex chars from a SHA-256 hash of the normalised URL).
  4. All curated entries are marked is_grant_maker = true with high confidence, because they were hand-verified.
Decision: Merge into register records — When a curated funder matches an existing register charity by hostname, the curated URL enriches the register record rather than creating a duplicate. This prevents the same organisation from appearing twice.
Step 01c

Discover from Grant Directories

File: scripts/pipeline-v3/01c-discover-directories.ts   Tavily API Paid

What happens

  1. Fetches content from 12 known grant directory websites using Tavily (to handle JS-rendered pages).
  2. Extracts all URLs found on those pages.
  3. Checks each URL against existing charities in the database (by hostname). If the hostname is new, inserts a new charity row with source = 'discovered'.
Assumption: These 12 directory sites are comprehensive and up-to-date. If a directory removes a listing, the charity is not removed from our database (no deletion logic exists).
Step 01d

Discover from Search

File: scripts/pipeline-v3/01d-discover-search.ts   Serper API Paid

What happens

  1. Runs ~50 Google search queries via the Serper API (Google search results as JSON).
  2. Queries include national terms ("NZ charitable trust funding applications"), per-region terms ("Canterbury community grants funding apply"), and per-sector terms ("NZ mental health community grants apply").
  3. Results are filtered to exclude known non-charity domains (news sites, social media, crowdfunding, grant directories themselves, business directories, professional services).
  4. New hostnames not already in the database are inserted as source = 'discovered'.
Decision: Excluded domains — An explicit blocklist prevents importing news articles, Wikipedia pages, LinkedIn profiles, etc. as "charities". The blocklist includes ~30 domains. If a legitimate funder has a presence only on a blocked domain, it will be missed.

5. Phase 2 — Website Validation

Goal: verify that website URLs actually belong to the charity

Step 02a

Validate Existing Websites

File: scripts/pipeline-v3/02a-validate-websites.ts   Raw Fetch Tavily API Paid (Tavily fallback)

Concurrency: 20 charities in parallel.

What happens for each charity with a website URL

  1. Domain blocklist check: If the website hostname is a known non-charity domain (social media, directories, professional services like law firms, trustee companies, email providers), the URL is cleared to NULL and status set to failed / blocked-domain. The charity keeps its row but loses its website.
    Full blocklist categories
    • Social media: google.com, facebook.com, linkedin.com, twitter.com, youtube.com, etc.
    • NZ government directories: charities.govt.nz, fundinginformation.org.nz, etc.
    • News: nzherald.co.nz, stuff.co.nz, rnz.co.nz, etc.
    • Crowdfunding: givealittle.co.nz, gofundme.com
    • Trustee companies: publictrust.co.nz, perpetualguardian.co.nz, etc.
    • Business directories: charitydata.co.nz, bizdb.co.nz, etc.
    • Generic email providers: gmail.com, hotmail.com, xtra.co.nz, etc.
    • Professional services (accounting, law): deloitte, kpmg, buddlefindlay, etc.
  2. HEAD check: Sends an HTTP HEAD request (15-second timeout). If the response is 404 or 410, the site is considered dead. 403 and 5xx responses are treated as "possibly blocked" (alive but can't confirm).
  3. Content fetch: Downloads the first 30,000 characters of the page via raw HTTP GET. If that fails, falls back to Tavily extraction (if API key is set).
  4. Validation: Checks if the page content proves this website belongs to this charity:
    • Registration number match (high confidence): Page contains the charity's CC##### registration number.
    • Exact name match (high confidence): Page contains the charity's full name (case-insensitive).
    • Title similarity (high confidence): The page's <title> tag matches the charity name with a Dice coefficient ≥ 0.6.
    • Core words match (medium confidence): Distinctive words from the charity name (excluding generic terms like "trust", "foundation", "community", "new", "zealand") all appear somewhere in the first 3,000 characters.
    • Weak similarity (low confidence): Dice coefficient ≥ 0.4 against the title.
    • No match: Status set to unvalidated.
Decision: Dice coefficient thresholds — Name similarity is measured using the Dice coefficient on character bigrams (pairs of adjacent characters). A score of 1.0 means identical bigram sets; 0.0 means no overlap. The threshold of 0.6 for title matching was chosen to allow minor variations (e.g., "Trust" vs "The Trust") while avoiding false positives. For full-content matching, 0.4 is used (weaker match is acceptable because the charity name may appear only once in a long page).
Assumption: Websites in the Charities Register are accurate. In practice, many are outdated, point to trustee company directories, or are generic law firm pages. The validation step catches some of these, but low-confidence validations may still let through wrong URLs.
Step 02b

Discover Websites from Email Domains

File: scripts/pipeline-v3/02b-discover-email-websites.ts   Free

For charities that have no website but do have an email address with a custom domain (not gmail, hotmail, xtra, etc.): uses the email domain as a candidate website URL. For example, info@brightfuture.org.nz → try https://brightfuture.org.nz.

Step 02c

Discover Websites via Registration Number Search

File: scripts/pipeline-v3/02c-discover-cc-search.ts   Serper API Raw Fetch Paid (cheap)

V4 NEW STEP: Added before the broader name-only search (02d) to find websites faster and with higher confidence. Charities found here skip the more expensive Step 02d entirely.

Concurrency: 5 in parallel.

Targets charities that still have no website after Steps 02a–02b, and have a CC-format registration number (register charities only — curated charities have synthetic CU numbers, not searchable).

What happens for each charity

  1. Search: Queries Serper (Google) for "{charity name}" "CC12345" (name in quotes + registration number in quotes), requesting 5 results, with the standard excluded domains list applied.
  2. Why this works: The CC registration number (e.g., CC12345) is a near-unique identifier. If a webpage contains both the charity's name and its registration number, it is almost certainly the charity's own website, or at minimum a page directly about that charity. The Charities Register itself is on the excluded domains list, so results from charities.govt.nz won't appear.
  3. Filter candidates: Removes results on blocked domains. Takes the top 3 remaining candidates.
  4. Validate each candidate: For each candidate:
    • Fetches the page content (raw HTTP GET, up to 30,000 chars)
    • Checks if the page contains the registration number (e.g., "CC12345") — this is the highest confidence validation signal (same as the "registration-number" method in Step 02a)
    • If the CC number is found on the page: accepts this URL as the charity's website with validated / search-cc-match status and stops
    • Falls back to the standard validateWebsite() check (name match, title similarity) if the CC number isn't found but the name matches
  5. If no candidate validates: The charity is left for Step 02d (broader name-only search).
Decision: CC number search runs before name-only search — This ordering is deliberate: the CC number search is more targeted (fewer false positives) and cheaper (one search call, strong validation means less need to check multiple candidates). Any charity found here is validated with high confidence and skips the broader, more ambiguous name-only search in Step 02d. This reduces total Serper API calls and page fetches downstream.
Assumption: A charity's own website will contain its CC registration number. This is true for many NZ charities (the Charities Act requires display), but not all — especially smaller organisations with basic websites. Charities not found here still get the name-only search in Step 02d as a fallback.
Step 02d

Discover Websites via Name Search

File: scripts/pipeline-v3/02d-discover-search-websites.ts   Serper API Raw Fetch Tavily API Paid

Concurrency: 5 in parallel (Serper rate-limit friendly).

Only searches for charities still without a website after Steps 02a–02c, that are likely grant-makers (purpose_to_give_grants = true, main_activity_id = 3, is_grant_maker = true, or medium/high classification confidence) and are not soft-excluded (schools, churches, sports clubs).

What happens for each charity without a website

  1. Search: Queries Serper (Google) for "{charity name}" New Zealand, requesting 5 results, with the standard excluded domains list applied (news, social media, directories, etc.).
  2. Filter candidates: Removes results on blocked domains (directories, professional services, email providers). Takes the top 3 remaining candidates.
  3. Validate each candidate (in order): For each of the 3 candidates:
    • Fetches the page content (raw HTTP GET, up to 30,000 chars; falls back to Tavily if raw fails)
    • Runs the same validateWebsite() check as Step 02a (registration number match, exact name match, title similarity via Dice coefficient, core-word match)
    • If validation succeeds with high or medium confidence: accepts this URL as the charity's website, marks it validated, and stops checking further candidates
    • If validation succeeds with low confidence only: accepts the URL but marks it as unvalidated / search-weak-match, and stops
    • If validation fails: tries the next candidate
  4. If no candidate validates: Marks the charity as no-website.
Decision: Top 3 candidates with validation — Rather than blindly accepting the top Google result, the step evaluates up to 3 candidates by actually fetching each page and verifying the charity's name or registration number appears in the content. This prevents assigning the wrong website when the top search result happens to be a directory listing or a news article about the charity rather than its own site. Low-confidence matches (Dice coefficient 0.4–0.6) are stored but flagged, so they can be treated with more caution downstream.
Assumption: 3 candidates are sufficient. If the charity's actual website is the 4th or 5th result (e.g., because directory listings or news articles rank higher), it will be missed. Increasing to 5 would add more fetch + validation calls but improve coverage.

6. Phase 3 — Grant Page Resolution

Goal: find the specific page on each funder's website where grant information lives

Step 03a

Resolve Curated Grant Pages

File: scripts/pipeline-v3/03a-resolve-curated.ts   Free

If a charity has a curated_grant_url (set during import from the 251-URL curated list), copy it directly to grant_page_url with source 'curated'. No fetching or validation needed — these URLs were hand-verified.

Step 03b

Navigate to Grant Pages (Nav Link Discovery)

File: scripts/pipeline-v3/03b-resolve-nav.ts   Raw Fetch Free

Concurrency: 30 charities in parallel (V4: increased from 10 — raw HTTP fetches have no rate limit).

What happens

  1. For each charity with a website but no grant_page_url: fetch the homepage HTML (raw GET, 15-second timeout).
  2. Parse all <a href="..."> links in the HTML. For each link, combine the href and the link text and check against a keyword regex: 
    /\b(grants?|funding|what.we.fund|apply|application|eligib|criteria|guidelines
|how.to.apply|deadlines?|rounds?|programmes?|opportunities|community.invest
|distributions?|what.we.support|p[uū]tea|tautoko|[aā]whina|tono|kaupapa)\b/i
    V4 NEW: Te Reo Māori keywords
    The regex now includes Te Reo Māori terms commonly used for grant/funding pages on bilingual NZ websites: pūtea (funds/finance), tautoko (support), āwhina (help/assistance), tono (apply/application), kaupapa (programme/initiative). This ensures the nav crawler recognises grant pages on Iwi trusts, Māori health providers, and other organisations that use Te Reo-first or bilingual navigation. Both macron and non-macron variants are matched (e.g., pūtea and putea).
  3. Filter out donation links: Links matching donation keywords (donate, donation, give now, support us, fundrais, make a gift, contribute) are excluded UNLESS the link also mentions "grants".
  4. Sort by relevance: Links found inside <nav> or <header> blocks rank higher (these are main navigation links, more likely to lead to important pages).
  5. Take the best match: The top-ranked link becomes grant_page_url with source 'nav-discovery'.
  6. V4 NEW: Two-level crawl for nested navigation
    If no grant-keyword links are found on the homepage, a second-level crawl is performed. Common "parent menu" links on the homepage that could contain grant sub-pages are followed one level deeper:
    • Links matching patterns like: what we do, about us, our work, community, support, services, programmes, invest, get involved, our impact
    • Only same-domain links are followed (no external sites)
    • Up to 5 intermediate pages are fetched (raw GET, 15-second timeout each)
    • Each intermediate page is parsed for grant-keyword links using the same regex as above
    • If a grant-keyword link is found on any intermediate page, it becomes grant_page_url with source 'nav-discovery-deep'
    Example: Homepage → "What We Do" (no grant keywords, but matches parent-menu pattern) → fetch that page → find "Our Grants" link → use that as the grant page.
  7. If still no grant links found: The homepage URL itself becomes the grant_page_url with source 'homepage-fallback'.
Decision: Homepage fallback — If no grant-related nav links are found even after the two-level crawl, we use the homepage rather than giving up. This is a trade-off: the homepage rarely contains detailed grant information, so extraction (Step 05a) will often find nothing. But some funders do have grant info directly on their homepage.
Decision: Intermediate page patterns — The set of parent-menu patterns (what we do, about us, our work, etc.) was chosen to cover common NZ charity website structures without being so broad that every link gets followed. Following up to 5 intermediate pages keeps the cost reasonable (free raw fetches) while catching the most common two-level navigation patterns. The patterns deliberately exclude links like "News", "Events", "Contact", "Donate" which are unlikely to lead to grant pages.
Assumption: Grant pages are reachable within two clicks from the homepage. Funders with three or more levels of navigation nesting (e.g., Home → About → Governance → Funding → Apply) would still be missed. Step 03c (Google site-search) serves as a fallback for these cases.
Step 03c

Search for Grant Pages

File: scripts/pipeline-v3/03c-resolve-search.ts   Serper API Paid

What happens

For charities that fell through to homepage-fallback AND are likely grant-makers (have positive classification signals): search Google with site:{hostname} grants OR funding OR "apply for". Takes the top result on the same domain that isn't a PDF. If a better page is found, updates grant_page_url with source 'search'.

Decision: Only search for likely grant-makers — The query filter requires is_grant_maker = true OR purpose_to_give_grants = true OR main_activity_id = 3. This avoids wasting Serper API calls on charities unlikely to have grant pages.

7. Phase 4 — Classification

Goal: determine which charities are actually grant-givers (vs. charities that only receive grants or provide services)

Step 04a

Rule-Based Pre-Classification

File: scripts/pipeline-v3/04a-classify-rules.ts   Free

Six rules applied in order. Each only affects charities where is_grant_maker IS NULL (not yet classified), except Rule 3 which can override a previous false classification.

RuleConditionResultConfidence
1 purpose_to_give_grants = true
(from Charities Register field PurposeToGiveGrantsAndDonations)		 Grant-maker = YES High
1B generates_funds_to_others = true
(from Charities Register field OrganisationGeneratesFundsGrantsDonationsToOthers)		 Grant-maker = YES High
2 main_activity_id = 3
(Charities Register activity code for "making grants/loans")		 Grant-maker = YES High
3 source = 'curated' OR curated_grant_url IS NOT NULL
(manually verified by us)		 Grant-maker = YES
(overrides even false)		 High
4 Name contains "gaming trust" or "gaming foundation" Grant-maker = YES High
5 Name contains: "grants fund", "grant fund", "community grant", "grants trust", "distribution trust/committee", "funding trust", "community fund", "energy trust", "licensing trust" Grant-maker = YES Medium
6 excluded_pattern = true AND no positive signals (purpose_to_give_grants not true, main_activity_id != 3, no curated URL) Grant-maker = NO Medium
Decision: Rule priority — Rules 1–3 use government-provided data fields, so they have "high" confidence. Rule 3 (curated) can override a prior false classification because we trust our own manual verification over automated rules. Rule 6 is negative classification — it removes charities with excluded-pattern names (schools, churches, sports clubs) from consideration, but only if they have no other grant-making signals.
Step 04b

Page-Content Classification

V4: REMOVED
This step used a blunt keyword-counting system (count "giving" keywords vs "receiving" keywords, classify based on which score is higher). It is removed in V4 for two reasons:
  1. False negatives are permanently destructive: If this step incorrectly classifies a charity as is_grant_maker = false (e.g., because their annual report page mentions "donations" and "support us" more than "apply for funding"), that charity never reaches extraction in Step 05a. Its grants are silently and permanently lost.
  2. The cost saving is negligible: Step 04c (GPT classification) processes charities in batches of 40 with name, full purpose text, and website content. At ~$10 for the entire run, sending the 04b charities directly to GPT instead is cheap and far more accurate than keyword counting.
All charities that 04b previously handled now flow directly to Step 04c (GPT classification).
Step 04c

GPT Classification

File: scripts/pipeline-v3/04c-classify-gpt.ts   OpenAI GPT-4o Paid

V4 FIX: Auto-classify charities with grant page URLs
Before calling GPT, any charity that already has a grant_page_url is automatically classified as is_grant_maker = true with classification_confidence = 'high' and classification_notes = 'rule:has-grant-page'. The reasoning: if Steps 03a–03c successfully discovered a grant page for this charity, that is already strong evidence they are a grant-maker — there is no need to spend GPT tokens confirming it. This removes these charities from the GPT classification batch entirely.

What happens

  1. Pre-filter: Charities with a grant_page_url are auto-classified as grant-makers (see V4 fix above) and excluded from GPT processing.
  2. Selects all remaining still-unclassified charities from the register source (those without a grant page URL).
  3. Sends them to GPT-4o in batches of 40, with 3 batches processed concurrently (V4: was sequential, now parallel for faster throughput).
  4. Each charity is represented with: 
    ID:123 | Name: Bay Trust
Purpose: To provide grants and funding for community organisations in the Bay of Plenty region...
Website: baytrust.org.nz
--- Website content (first 2,000 chars) ---
Bay Trust is one of the largest funders in the Bay of Plenty...
---
  5. GPT returns a JSON array with is_grant_maker (true/false/null), confidence (high/medium/low), and notes.
  6. Protection for rule-based classifications: The database UPDATE has a WHERE clause that never overwrites a high-confidence rule-based classification. This prevents GPT from contradicting verified signals.
V4 FIX: Richer GPT input for better classification
V3 sent GPT a truncated 200-character purpose snippet and two URLs — no actual webpage content. This meant GPT was often guessing based on a charity name and a sentence fragment. V4 makes two improvements:
  1. Full purpose text: The purpose field is sent in its entirety instead of being truncated to 200 characters. The register purpose field is typically 1–3 paragraphs, so the cost impact is negligible.
  2. Website content snippet: For charities that have a website_url, the first 2,000 characters of the homepage content are fetched and included in the GPT prompt. This gives GPT actual evidence about what the charity does, rather than relying on the name and a short purpose string. Charities without a website still get classified on name + purpose alone.
The website content is fetched in bulk before the GPT batches start (raw HTTP GET, same-domain only, no external API cost). Failed fetches are silently skipped — the charity is still classified, just with less context.

8. Phase 5 — Grant Extraction

Goal: extract structured grant data from funder web pages using GPT-4o-mini

Step 05a

Extract Grants (Tavily/Fetch + GPT-4o-mini)

File: scripts/pipeline-v3/05a-extract-tavily.ts   Tavily API Raw Fetch OpenAI GPT-4o-mini Paid (~$15)

Concurrency: 40 funders in parallel (V4: increased from 20 — GPT-4o-mini has much higher rate limits than GPT-4o).

This is the most important step. It processes every classified grant-maker that has a grant page URL, extracts the grant information from the page content using GPT-4o-mini, validates the results, and stores them in the database.

V4 CHANGE: GPT-4o-mini instead of GPT-4o
V3 used GPT-4o (~$90 GPT cost). V4 switches to GPT-4o-mini (~$5–6 GPT cost) — a ~94% cost reduction. GPT-4o-mini has the same 128k context window and supports JSON mode, but is less capable at nuanced judgment calls (money-flow direction, one-programme rule, attribution guard). The pipeline's post-extraction validation (name checking, ownership filtering, quality scoring, dedup) acts as a safety net for any quality loss. If quality proves insufficient on a test run (--limit 30), the model can be upgraded back to GPT-4o for the full run.

Funder selection

Query selects charities where:

  • is_grant_maker = true
  • grant_page_url IS NOT NULL
  • enriched_at IS NULL (not yet processed, unless --force flag is used)

Ordered by priority: curated sources first, then validated websites, then unvalidated.

V4 NEW: URL-Level Dedup Before Extraction
After selecting funders, they are grouped by normalised grant_page_url (protocol and www prefix stripped, lowercased, trailing slashes removed). If multiple funders share the same URL (e.g., different trust registrations managed by the same law firm), only one canonical funder is processed. The others are marked as enriched_at = NOW() and skipped.

Canonical funder selection: prefer curated source → prefer CC-numbered register charity → lowest ID (stable tiebreaker).

Why: Without this, the same page would be sent to GPT once per funder sharing that URL, wasting API calls and producing duplicate grants attributed to the wrong funders.

Page content fetching (per funder)

  1. PDF grant page handling: If the grant page URL ends in .pdf:
    V4 FIX: Extract PDF content directly
    Instead of falling back to the homepage (which loses all the PDF content), the PDF is downloaded and its text extracted using pdf-parse. The extracted text (up to 15,000 chars) is included in the combined content sent to GPT. The homepage is still used for nav/sub-page discovery alongside the PDF content.
    If the PDF cannot be downloaded or parsed, the pipeline falls back to the homepage as before.
  2. Main page fetch: Uses Tavily extraction if API key is set (handles JS-rendered pages, 403-blocked pages), otherwise falls back to raw HTTP GET. Returns up to 80,000 characters of page content. Timeout: 15 seconds.
  3. Sub-page discovery: The main page's HTML is parsed for navigation links matching grant-related keywords (same regex as Step 03b, including Te Reo terms). Up to 3 additional pages are fetched, each limited to 15,000 characters.

    Sub-page content is concatenated with markers:

    --- Page 1: https://example.org/funding ---
[main page content, up to 80,000 chars]

--- Page 2: https://example.org/funding/eligibility ---
[sub-page content, up to 15,000 chars]

--- Page 3: https://example.org/funding/apply ---
[sub-page content, up to 15,000 chars]
  4. V4 NEW: PDF link detection and extraction
    After fetching the main page and sub-pages, the HTML is scanned for links to .pdf files whose link text or filename contains grant-relevant keywords: guideline, criteria, eligibility, application, how to apply, funding guide, grant info, information sheet.

    Up to 3 matching PDFs are downloaded and text extraction is attempted in two stages:
    1. Text-based PDFs: pdf-parse extracts text directly. This handles the majority of modern digital PDFs at zero API cost.
    2. Scanned-image PDFs (fallback): If pdf-parse returns minimal text (< 50 chars) but the PDF has pages, it’s likely a scanned image. The PDF pages are rendered to images using pdfjs-dist (Mozilla’s PDF.js) and sent to GPT-4o-mini Vision for text extraction. Up to 5 pages per PDF, using detail: ‘low’ for minimal cost (~$0.01–0.03 per scanned PDF).
    Each PDF's extracted text is limited to 15,000 characters and appended to the combined content: 
    --- PDF: https://example.org/funding/guidelines-2024.pdf ---
[extracted PDF text, up to 15,000 chars]
    Limits: max 3 PDFs per funder, max 15,000 chars per PDF, max 10MB file size (skip large annual reports), max 5 pages per scanned PDF for Vision OCR, 20-second download timeout per PDF.

    Why: Many traditional NZ trusts and councils provide grant details (eligibility criteria, amounts, deadlines, application instructions) solely in downloadable PDFs. Without extracting this content, GPT sees a page that says "Download our guidelines" but has no actual guidelines — resulting in low-quality or zero-grant extractions.

    New dependencies: pdf-parse (text extraction, ~50KB), pdfjs-dist (PDF page rendering, pure JS), canvas (Node.js canvas for pdfjs-dist rendering, has native bindings).
  5. Total content sent to GPT: The GPT prompt truncates to 80,000 characters. Content is assembled with reversed priority to ensure the most valuable data is never truncated:
    V4 FIX: Reversed truncation priority
    V3 concatenated content as: main page → sub-pages → PDFs, then truncated at 80k. This meant a 60k marketing-heavy main page would leave only 20k for the sub-pages and PDFs that contain the actual grant details (eligibility, criteria, amounts).

    V4 reverses the order: PDFs first (criteria, guidelines) → sub-pages second (eligibility, deadlines) → main page fills the remainder. The main page is guaranteed at least 10,000 characters (for funder name, overview, context), but PDFs and sub-pages always get their full allocation.
Assumption: 3 sub-pages and 3 PDFs are enough to capture all grant information. Funders with more than 6 supplementary resources will have some missed. The sub-pages and PDFs are selected by keyword relevance, so the most important ones should be prioritised, but there is no guarantee.
Assumption: GPT-4o-mini Vision can reliably read text from scanned PDF pages at detail: 'low' resolution. Very poor quality scans (faded photocopies, handwritten annotations, heavily skewed pages) may still produce unreliable text. The expected volume of scanned-image grant PDFs in NZ is very small — most are digital.

Keyword pre-filter

Before sending content to GPT (which costs money), a quick keyword check is performed:

  • Must contain at least one of: grant, fund, funding, community support, application, apply, programme, distribution, eligib
  • Must not look like a donation page: If the content matches donation keywords (donate, donation, support us, give now, fundraising, make a gift) AND does NOT match grant-giving override keywords (we fund, we offer, apply for, application, eligib, grant program, funding available, open for, distribution, community fund, funding round) → skip

If the content fails this check, the funder is marked with status no-grants and NOT sent to GPT.

Decision: Keyword pre-filter — This saves GPT API costs by not sending clearly irrelevant pages. But it means pages that use unusual language for grants (e.g., "sponsorship", "investment", "assistance") could be skipped.

The combined page content is sent to GPT-4o-mini with a detailed prompt. Here is what the prompt tells GPT:

Key instructions in the user prompt:

  1. Direction of money flow: GPT is told to ONLY extract grants that this organisation GIVES OUT. If the page is about donating TO the organisation, receiving grants, or providing services — return empty results.
  2. Funder context provided:

  6. V4 NEW: Closed Rounds Rule

Only return empty results if the programme has been permanently discontinued."

    Why: Many NZ grants operate on annual or biannual cycles (e.g., close March 31, reopen September). If the pipeline runs during a closed period, GPT would otherwise see "applications are now closed" and return empty results, silently losing the opportunity.

Post-extraction validation (validateAndEnrich)

After GPT returns its JSON, each extracted grant goes through validation before being inserted into the database:

  1. Grant name hallucination check: The grant name must appear in the actual page content (case-insensitive, with plural/singular and programme/program variants). If all words in the name are generic (grant, fund, programme, the, a, for, etc.) and the name is short, it is accepted anyway — many NZ funders have unnamed programmes. If the name contains distinctive words not found in the page, the grant is filtered out with a warning log.
  2. V4 NEW: Funder ownership check — If the grant has an attributed_funder field, its similarity to the actual funder name is checked using the Dice coefficient. If the similarity is below 0.4 (very different name), the grant is filtered out. This catches cases where GPT extracted a grant belonging to another organisation mentioned on the page.
  3. Field sanitisation:
    • Regions: only valid NZ region IDs are kept; invalid values are removed
    • Sectors: only valid sector IDs are kept
    • Type: must be one of the 6 valid enum values; defaults to "Other"
    • Round frequency: must be one of the 5 valid enum values; invalid → null
    • Amounts: rounded to integers
  4. Form URL validation: The application form URL must be on the same domain as the source page (or a subdomain), OR be on a known trusted portal domain (SmartyGrants, Fluxx, SurveyMonkey, Google Forms, Typeform, JotForm, Microsoft Forms). URLs that fail both checks are discarded (set to null).
  5. Individual-only detection: Checks grant name, eligibility, and description for patterns like "scholarship", "bursary", "fellowship", "must be a student", "for individuals". Flags the grant as individual_only = true.
  6. Grant ID generation: A deterministic ID is computed: 'g_' + SHA-256(funder_name | grant_name | page_url) truncated to 16 hex characters. This means re-running extraction for the same funder/grant/URL produces the same ID, enabling safe upserts.
  7. Quality score (0–100): Based on field completeness:
    Description > 50 chars+20
    Eligibility ≥ 2 items+15
    Amount max present+15
    Deadline present+15
    Application form URL+10
    Sectors ≥ 1+10
    Regions (present or null = national)+10
    Key contacts+5
  8. Field confidence metadata: Each field is tagged as extracted (from the page), inferred (from the funder's data, e.g., regions inherited from the charity record), or default.

Database insertion

Uses INSERT ... ON CONFLICT (id) DO UPDATE. The deterministic grant ID ensures that re-running extraction updates existing grants rather than creating duplicates. Each grant stores:

  • funder_id — links to the charities table
  • source_url — the specific page the grant was extracted from
  • extraction_pages — array of all URLs that were fetched and sent to GPT (main + sub-pages)
  • extraction_model — "gpt-4o-mini"
  • discovery_method — e.g., "curated-tavily", "nav-discovery-fetch"
  • pipeline_version = 4

After successful insertion, the charity's enriched_at is set to NOW().

V4 NEW: Deactivate superseded grants
After successful extraction (at least 1 grant found), any of the funder's existing active grants that were NOT in the current extraction batch are deactivated with scrape_notes = 'extract:superseded'. This prevents discontinued or renamed grants from persisting forever in the database.

Guard: Only runs when extraction produced ≥ 1 grant. If extraction returned 0 grants (fetch-failed, gpt-error, no-grants), the funder's existing grants are left untouched — a transient failure should not wipe out real data.
Assumption: enriched_at is only set when at least 1 grant is found. If extraction produces 0 valid grants (keyword filter, GPT returns empty, all filtered by validation), the funder stays with enriched_at = NULL for statuses fetch-failed and gpt-error, but gets marked done for no-grants. This means some funders can get permanently stuck if they repeatedly fail.

9. Phase 6 — Deduplication & Cleaning

Goal: remove duplicate, mis-attributed, and historical grants

Step 06a

Grant Deduplication (6 Passes)

File: scripts/pipeline-v3/06a-dedup.ts   OpenAI GPT-4.1-mini (Pass 3 only) Mostly free

Loads ALL active V3 grants into memory, then runs 6 sequential passes. Each pass deactivates grants by setting is_active = false (they remain in the database for audit purposes).

Pass 1: Exact Dedup

  • Groups by: funder_id + source_url
  • Threshold: Name similarity ≥ 0.85 (Dice coefficient)
  • Action: Keeps the record with the higher completeness score, deactivates the other
  • Purpose: Catches true duplicates where the same grant was extracted twice from the same page for the same funder

Pass 1B: Cross-Funder Contamination

  • Groups by: Normalised source_url (strips protocol, www, trailing slashes, lowercased) across ALL funders
  • Detects: When 2+ different funders have grants from the same URL
  • Threshold: Name similarity ≥ 0.8 across funders
  • Resolution: Keeps the grant whose funder_name best matches the grant name (highest Dice similarity). Deactivates the rest.
  • Purpose: When multiple charities share a grant page (e.g., law firm managing multiple trusts), GPT extracts the same grant for each. This pass keeps only the correct attribution.
V4 context: The URL-level dedup in Step 05a now prevents most cross-funder contamination at the source. Pass 1B serves as a safety net for any cases that still slip through.

Pass 2: Name Similarity Dedup

  • Groups by: funder_id only
  • Threshold: Name similarity ≥ 0.85 (different types) or ≥ 0.90 (same type)
    V4 FIX: Raised threshold for same-type grants from 0.85 to 0.90
    At 0.85, grants like "Community Health Fund" and "Community Housing Fund" (Dice ~0.87, both type "Community") would be incorrectly merged. At 0.90, these are preserved while true duplicates like "Community Health Grant" vs "Community Health Grants" (Dice ~0.96) are still caught.
  • Guard: If both grants have different type values (e.g., one is "Community", the other is "Government"), they are considered distinct and not merged
  • Action: Keeps the more complete record, deactivates the other
  • Purpose: Catches slightly different names for the same programme (e.g., "Community Grants" and "Community Grant Programme")

Pass 3: False-Split Audit (GPT)

  • Targets: Groups of 3–8 grants from the same funder and same source URL
  • Tool: GPT-4.1-mini (smaller, cheaper model) in batches of 5 groups
  • Verdict: "false_split" (one programme incorrectly split into multiple records) or "distinct" (genuinely separate programmes)
  • Signals for false-split: Theme-word names, no distinct eligibility, no distinct forms, overlapping descriptions
  • Signals for distinct: Different forms, different amounts, explicitly separate streams, different deadlines
  • When in doubt: Classify as "distinct" (preserve data)
  • Only runs if OPENAI_API_KEY is set

Pass 4: Misnamed Grants

  • Detects: Grants where the grant name is nearly identical to the funder name (similarity ≥ 0.85)
  • Action: Renames (does NOT deactivate) to "{Funder Name} - General Fund"
  • Purpose: When GPT couldn't find a specific programme name and used the funder name, this makes it more descriptive

Pass 5: Historical Round Listings

  • Pattern: /\b(successful|approved|awarded|past|previous|completed)\s+(grants?|applications?|recipients?|round)/i
    V4 FIX: Removed "closed" from the pattern. A "closed round" or "closed grants" typically means the current round is between cycles, not that the grant is a historical listing. Removing it prevents deactivation of active recurring grants during their closed period.
  • Checks: Grant name only
    V4 FIX: No longer checks description
    V3 checked both grant name and description, which caused false positives. Descriptions routinely reference past rounds as context for current ones — e.g., "Previous round closed March 2024. Next round opens June 2024." This would incorrectly deactivate an active, recurring grant. The pattern is now only applied to grant names, where matches like "Successful Grants June 2023" are reliably historical listings.
  • Action: Deactivates — these are records of past grants, not current opportunities

Has sectors+2
Has regions+3
Has eligibility+2
Has amount_min+1
Has amount_max+1

Step 06c

Verify & Score

File: scripts/pipeline-v3/06c-verify-and-score.ts   Free

What happens

  1. URL verification: HEAD requests to all grant URLs and application form URLs (concurrency 25). Marks broken URLs.
  2. Deadline management: Finds grants with ISO-date deadlines that have passed:
    • Recurring grants (is_recurring = true or has a round_frequency): the date is converted to a text schedule — e.g., "annual - typically March"
    • Non-recurring grants: deactivated after the grace period
    V4 FIX: More conservative deadline deactivation
    V3 only preserved grants if is_recurring = true OR round_frequency was specifically "annual" or "biannual". Grants with round_frequency = 'quarterly' or 'rolling' would be deactivated. V4 fixes this:
    • Any round_frequency value (annual, biannual, quarterly, rolling, irregular) now triggers conversion to a text schedule instead of deactivation
    • Grace period extended from 30 days to 90 days for genuinely non-recurring grants, allowing more time for a re-scrape to confirm the grant is truly gone
    • Only deactivates when both is_recurring is falsy AND round_frequency is null AND 90+ days past deadline
    Why: Many NZ grants operate on cycles (close March 31, reopen September). If GPT sets is_recurring = false while still providing a round_frequency, V3 would deactivate the grant after 30 days. V4 is more conservative — if there is any signal that the grant recurs, it is preserved.
  3. V4 NEW: Grant TTL — re-scrape stale funders
    Funders whose enriched_at is older than 180 days have it reset to NULL, flagging them for re-extraction on the next pipeline run. This prevents grants from going permanently stale — especially important for rolling grants with no ISO deadline that would otherwise never trigger deadline management.

    The grants themselves stay active until re-extraction confirms or replaces them (the ON CONFLICT DO UPDATE upsert handles this safely). If re-extraction returns 0 grants, the old grants remain because they have different IDs.
  4. Final quality scores: Recomputes 0–100 quality scores for all active grants and generates summary statistics.

10. Phase 7 — Charity Dedup

Goal: merge duplicate charity records that represent the same real-world organisation

Step 07

Deduplicate Charities

File: scripts/pipeline-v3/07-dedup-charities.ts   Free

Current (V3) behaviour

Groups charities by LOWER(TRIM(name)) — exact name match only (case-insensitive). For each group:

  1. Pick winner: Prefer CC-numbered register entry → most active grants → has curated URL → has website → has been enriched → lowest ID
  2. Re-point grants: All grants from losers are updated to point to the winner's funder_id
  3. Merge fields: Best website, grant page, email, city, curated URL, funder type are copied to the winner. Boolean flags (is_grant_maker, purpose_to_give_grants) are OR'd across the group.
  4. Delete losers: Loser charity rows are hard-deleted from the database
V4 NEW: Enhanced Charity Dedup
Two additional dedup strategies are added before the existing name-exact dedup:
  • URL-based dedup (highest confidence): Charities sharing the same normalised website URL or grant page URL are grouped and merged using the same winner-selection logic.
  • Fuzzy name dedup: Charities with name similarity ≥ 0.85 (Dice coefficient) are candidates for merge, but only auto-merged if they also share a domain, email domain, or grant page URL. Name-only fuzzy matches without a URL/email signal are logged as candidates for manual review (not auto-merged).
Decision: Hard deletion of losers — Loser charity rows are permanently deleted (not soft-deleted). This is safe because all grants have been re-pointed and the loser had no unique data. But it means the merge cannot be undone without restoring from a database backup.

Appendix: Summary of V4 Architectural Changes

ProblemV3 Approach (reactive)V4 Approach (preventive)
Linked-funder leakage
GPT extracts grants from other organisations mentioned on the page		 No prevention. Caught inconsistently by name validation. 1. Prompt tells GPT to exclude other funders' grants
2. GPT must name the attributed_funder per grant
3. Post-extraction check: if attributed_funder doesn't match the funder being processed (Dice < 0.4), the grant is filtered out
Cross-funder contamination
Multiple charities share the same grant page URL		 All funders processed independently; duplicates caught by dedup Pass 1B after extraction 1. Before extraction: group funders by normalised URL
2. Pick one canonical funder per URL; skip the rest
3. Pass 1B remains as safety net
Duplicate charity records
Same org with multiple registrations		 Only exact name match (case-insensitive) 1. URL-based dedup (same website = same org)
2. Fuzzy name dedup (Dice ≥ 0.85) with URL/email confirmation
3. Name-only matches logged for manual review
PDF content gap
Grant details locked in PDF downloads, invisible to HTML extraction		 PDFs ignored entirely. If grant_page_url is a PDF, falls back to homepage. 1. PDF grant pages: download and extract text directly instead of falling back
2. PDF links on grant pages: scan HTML for PDF links with grant keywords, download up to 3, extract text
3. Scanned-image PDFs: GPT-4o-mini Vision fallback renders pages to images and reads the text
4. PDF text appended to combined content before GPT processing
Te Reo Māori navigation
Grant pages labelled in Te Reo missed by English-only keyword regex		 English keywords only in nav crawler Added Te Reo Māori terms to nav-link regex: pūtea, tautoko, āwhina, tono, kaupapa (with macron variants)
Stale/discontinued grants
Grants persist forever even after funder removes them		 No cleanup mechanism. Old grants remain active indefinitely. 1. After extraction: deactivate superseded grants not in current batch
2. Grant TTL: funders not re-scraped in 180 days flagged for re-extraction
3. Grace period for expired deadlines extended from 30 to 90 days
Closed-round data loss
GPT skips grants showing "applications closed"		 No guidance to GPT. Between-rounds grants silently dropped. 1. Prompt: "Closed Rounds Rule" — must extract closed-but-recurring grants
2. Dedup: "closed" removed from historical pattern regex
3. Deadline management: any round_frequency preserves the grant
Content truncation
PDFs and sub-pages truncated when main page is large		 Main page first, sub-pages/PDFs after — often truncated at 80k Reversed priority: PDFs first → sub-pages → main page fills remainder (min 10k guaranteed)
