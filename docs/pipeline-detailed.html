1. Architecture Overview

GrantSearch NZ is a Next.js 16.1.6 App Router application that helps NZ non-profits find, evaluate, and track grant applications. The core innovation is a two-pass AI search pipeline that queries a pre-populated Postgres database of ~4,500 active grants, triages them cheaply, then scores the relevant ones in detail. The application is multi-tenant, scoping all runtime data by organisation (org_id), with authentication via NextAuth v5.

Tech Stack
Frontend
  • Next.js 16.1.6 (App Router) + React 19 + TypeScript
  • Tailwind CSS 4 + shadcn/ui (base-nova style)
  • Dark mode via ThemeProvider + ThemeToggle (class-based Tailwind dark variant)
  • SWR for client-side data fetching & caching
  • Vercel hosting
Backend / Services
  • Neon Postgres — serverless database (grants, charities, orgs, users, app storage)
  • NextAuth v5 — Credentials provider, JWT sessions, 30-day max age
  • Vercel Blob — document storage (application supporting documents)
  • OpenAI GPT-4.1-mini — triage & scoring (search), enrichment (scripts)
  • OpenAI GPT-4o — deep search analysis
  • Serper — web search (deep search only)
  • Tavily — page content extraction (deep search + scripts)

Key design principles: (1) The search pipeline has zero external search/scraping costs. All grant data lives in the DB. The only runtime cost is GPT-4.1-mini tokens for triage and scoring. Deep search (per-grant, user-triggered) is the only operation that calls Serper/Tavily. (2) All runtime data is scoped by org_id, enabling multi-tenancy — each organisation sees only its own saved searches, shortlists, applications, documents, and checklists.

3. Authentication & Multi-Tenancy

Authentication is handled by NextAuth v5 with a Credentials provider. Sessions use JWT strategy with a 30-day max age. All runtime data is scoped by org_id, derived from the authenticated user's organisation.

Auth Flow
  1. Signup (/signup) — User provides name, email, password, and organisation name. Creates organisation + user in a single transaction. Redirects to login.
  2. Login (/login) — Email/password authentication. On success, JWT session includes userId and orgId.
  3. Invite (/invite/[token]) — Existing team members send invitations via Settings. Invitee clicks link, sets password, joins the existing organisation.
  4. Session — Every API route calls getOrgId() (from lib/auth-helpers.ts) to extract the org_id from the JWT. All DB queries are scoped by this org_id.
Multi-Tenancy Model
  • Every runtime storage table uses (org_id, ...) composite primary keys
  • All API endpoints derive org_id from the session JWT via getOrgId()
  • No cross-org data leakage is possible — queries always include WHERE org_id = $1
  • Pipeline/reference tables (charities, grants) are shared across all organisations (global grant database)
  • Dark mode preference is persisted client-side via ThemeProvider (class-based Tailwind dark variant)

4. Search API — POST /api/search

File: app/api/search/route.ts. Response is streamed via SSE (Server-Sent Events) for real-time progress updates. All data scoped by org_id.

Request (OrgInfo)

{
  searchTitle: string,     // user-provided title for this search
  website: string,         // org website URL
  linkedin?: string,       // org LinkedIn URL
  fundingPurpose: string,  // what funding is for
  fundingAmount: number,   // NZD amount sought
  market: string,          // 'nz' | 'au'
  regions: string[],       // region IDs the org operates in
  sectors: string[],       // sector IDs (e.g. ['health', 'youth'])
  orgType: string,         // 'registered-charity' | 'charitable-trust' | ...
  previousFunders?: string // free text
}

Pipeline

Step 1: DB Query — searchGrants(sectors, regions) queries the grants table with:

  • Region filter: empty regions = nationwide (no filter); non-empty = overlap match (&& operator)
  • Grants with NULL/empty regions always included (nationwide grants)
  • Order: sector overlap count DESC, funder name ASC
  • Returns all active grants matching (typically 500–4,500)

Step 2: Pass 1 — Triage (GPT-4.1-mini)

  • All grants split into batches of 100
  • Each batch sent to GPT-4.1-mini with the RELEVANCE_TRIAGE_PROMPT
  • Model classifies each grant as RELEVANT or SKIP
  • SKIP only when ≥90% confident: wrong sector, wrong region, wrong org type, dramatic amount mismatch, or non-grant content
  • On error, defaults all to RELEVANT (never drops grants due to API failure)
  • Concurrency: 15 parallel batches
  • Typically filters to 200–400 relevant grants

Step 3: Pass 2 — Scoring (GPT-4.1-mini)

  • Relevant grants split into batches of 25
  • Each batch sent with SCORING_SYSTEM_PROMPT + org context
  • Model scores on 3 dimensions: alignment (0–10), ease (0–10), attainability (0–10)
  • Overall = (alignment × 0.5) + (attainability × 0.3) + (ease × 0.2)
  • First batch also generates orgSummary (2–3 sentence org description)
  • Concurrency: 15 parallel batches
  • Includes multi-check logic: URL quality, geographic eligibility, form-of-support match, regional relevance, deadline validation

Step 4: Filter & Stream

  • Remove grants with alignment < 5.0
  • Assign deterministic IDs: g_ + sha256(funder|name|url)[:16]
  • Stream SearchResult via SSE with progress events for each pipeline stage

Response (SearchResult, streamed via SSE)

{
  grants: GrantOpportunity[],  // scored grants
  orgSummary: string,          // AI-generated org description
  searchedAt: string,          // ISO timestamp
  market: string,              // 'nz'
  inputs?: OrgInfo             // echoed for re-use
}

5. Deep Search API — POST /api/deep-search

File: app/api/deep-search/route.ts

Triggered per-grant by the user. Performs live web research to find precise details the DB may lack.

Phase 1: Web Search (Serper)

  • 6–8 parallel search queries targeting: application forms, guidelines, deadlines, eligibility, past recipients
  • Site-specific searches on the funder's own domain
  • Collects unique URLs, prioritises funder's own domain

Phase 2: Content Extraction (Tavily)

  • Up to 10 discovered URLs extracted via Tavily
  • Full page content (bypasses JS-rendered pages, 403s)
  • Funder's own domain URLs prioritised

Phase 3: AI Analysis (GPT-4o)

  • All extracted content sent to GPT-4o (full model, not mini) for structured analysis
  • Extracts: precise amounts, open/close dates, eligibility criteria, application form URL & type, checklist of required/optional documents, key contacts, past recipient insights
  • Checklist items include: item name, description, whether required or optional
  • Recalibrates all three scores (alignment, ease, attainability) with reasoning for each change

Phase 4: DB Write-back

  • Discovered fields written back to the grants table via writeDeepSearchUpdates()
  • Only fills NULL fields — existing data is never overwritten
  • Fields: application_form_url, amount_min, amount_max, deadline, checklist (JSONB)
  • Checklist data stored as DeepSearchChecklistItem[] in the grants table for future searches
  • Benefits all future searches (not just the user who triggered deep search)

Response (DeepSearchResult)

{
  grantId, grantName, funder, grantUrl, searchedAt,
  amountMin?, amountMax?, amountNotes?,
  applicationOpenDate?, applicationCloseDate?, dateNotes?,
  checklist: DeepSearchChecklistItem[],
  applicationFormUrl?, applicationFormType?, applicationFormNotes?,
  eligibilityCriteria: string[],
  scores: { alignment, ease, attainability, overall },
  scoreChanges: { alignment: { old, new, reason }, ease: ..., attainability: ... },
  additionalInfo?, keyContacts?, pastRecipientNotes?,
  sourcesUsed: { url, title }[]
}

6. Storage APIs

All storage uses Neon Postgres (no localStorage). Tables created on first use. All endpoints scoped by org_id via getOrgId().

Endpoint Methods Description
/api/saved-searches GET, POST, DELETE Save/list/delete search results. Auto-saved after each search. Stores full SearchResult as JSONB. Scoped by org_id.
/api/shortlist GET, POST, DELETE Add/remove/list shortlisted grants. Batch check endpoint. Grouped by search title. Scoped by org_id.
/api/deep-search-results GET, POST, DELETE Store/retrieve deep search results. Batch lookup by grant IDs. Scoped by org_id.
/api/applications GET, POST, PUT, DELETE Full CRUD for application tracking. Status transitions, notes, financial amounts. Scoped by org_id.

7. Document Management APIs

Documents are stored in Vercel Blob with metadata in Postgres. Each document belongs to an organisation and can be categorised and attached to checklist items.

Endpoint Methods Description
/api/documents/upload POST Upload a document to Vercel Blob. Accepts multipart form data with file, category, and optional notes. Returns document metadata with blob URL.
/api/documents GET, POST, PUT, DELETE CRUD for document metadata. GET lists all documents for the org (filterable by category). PUT updates category/notes. DELETE removes blob and metadata.
/api/documents/download GET Proxied download of a document from Vercel Blob. Validates org ownership before serving.
Document Categories
Defined in lib/document-categories.ts. Categories include: financial statements, governance documents, identity/registration, project plans, letters of support, annual reports, and other. Each category has a label and optional description for the UI.

8. Checklist System APIs

Application checklists are derived from deep search results and allow users to track required documents and preparation items per application. Checklist items can have documents attached from the document library.

Endpoint Methods Description
/api/checklist GET, POST, PUT, DELETE CRUD for checklist items. GET returns items for a grant application (by grant_id). POST creates items (typically from deep search checklist data). PUT toggles checked status or updates item details. DELETE removes an item.
/api/checklist/attach POST, DELETE Attach or detach documents to/from checklist items. Creates/removes entries in the checklist_documents junction table.

14. Scoring System Details

Pre-checks (applied before scoring)

  1. URL Quality Check: If content is a news article, press release, or past-recipients list rather than a grant program page → all scores zeroed.
  2. Geographic Eligibility: If grant is restricted to a country/region that isn't NZ → alignment=0, attainability=0, overall=0.

Scoring Dimensions

Dimension Weight Factors
Alignment 50% Grant purpose vs org mission, form of support match (cash vs in-kind), sector overlap, specific funding request match
Attainability 30% Competition level, eligibility fit, regional relevance, org type match, amount range match
Ease 20% Application complexity: simple online form (9–10) vs multi-stage with site visits (1–2)

Formula: Overall = (alignment × 0.5) + (attainability × 0.3) + (ease × 0.2), rounded to 1dp

Filter: Only grants with alignment ≥ 5.0 are returned to the user.

Score Recalibration (Deep Search)

When deep search runs, GPT-4o recalibrates all three scores based on the detailed information discovered. Each change includes a reason explaining why the score moved (e.g. "Application requires detailed financials and site visit — reduced ease from 7 to 4").
