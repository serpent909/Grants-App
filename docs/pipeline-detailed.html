1. Architecture Overview

GrantSearch NZ is a Next.js 14 App Router application that helps NZ non-profits find, evaluate, and track grant applications. The core innovation is a two-pass AI search pipeline that queries a pre-populated Postgres database of ~4,500 active grants, triages them cheaply, then scores the relevant ones in detail.

Tech Stack
Frontend
  • Next.js 14 (App Router) + TypeScript
  • Tailwind CSS + shadcn/ui components
  • SWR for client-side data fetching & caching
  • Vercel hosting
Backend / Services
  • Neon Postgres — serverless database (grants, charities, app storage)
  • OpenAI GPT-4o-mini — triage & scoring (search), enrichment (scripts)
  • OpenAI GPT-4o — deep search analysis
  • Serper — web search (deep search only)
  • Tavily — page content extraction (deep search + scripts)

Key design principle: The search pipeline has zero external search/scraping costs. All grant data lives in the DB. The only runtime cost is GPT-4o-mini tokens for triage and scoring. Deep search (per-grant, user-triggered) is the only operation that calls Serper/Tavily.

2. Database Schema

Neon Postgres with connection pooling. Migrations in migrations/.

charities

Source-of-truth for funder identity. Populated from the NZ Charities Register and curated funders list.

id              INTEGER PRIMARY KEY
charity_number  TEXT            -- Charities Register number (null for curated-only)
name            TEXT NOT NULL
url             TEXT            -- website_url or homepage
charity_status  TEXT
primary_activity TEXT
source          TEXT            -- 'register' | 'curated'
curated_grant_url TEXT          -- hand-verified grants page URL
regions         TEXT[]          -- region IDs this funder serves
enriched_at     TIMESTAMPTZ    -- when grant programs were extracted
created_at      TIMESTAMPTZ
updated_at      TIMESTAMPTZ

grants

Individual grant programs, linked to funders. This is the table queried by the search API.

id                  TEXT PRIMARY KEY   -- deterministic: 'g_' + sha256(funder|name|url)[:16]
funder_id           INTEGER REFERENCES charities(id)
name                TEXT NOT NULL
type                TEXT               -- Government|Foundation|Corporate|Community|International|Other
description         TEXT
url                 TEXT               -- best known application/info URL
source_url          TEXT               -- page the grant was extracted from
amount_min          INTEGER            -- NZD
amount_max          INTEGER            -- NZD
regions             TEXT[]             -- region IDs (null/empty = national)
sectors             TEXT[]             -- sector IDs
eligibility         TEXT[]
deadline            TEXT               -- ISO date, 'rolling', 'annual - typically ...', etc.
is_recurring        BOOLEAN
round_frequency     TEXT               -- annual|quarterly|rolling|irregular
application_form_url TEXT
checklist           JSONB              -- DeepSearchChecklistItem[]
key_contacts        TEXT
last_scraped_at     TIMESTAMPTZ
last_verified_at    TIMESTAMPTZ
is_active           BOOLEAN DEFAULT true
scrape_notes        TEXT
created_at          TIMESTAMPTZ
updated_at          TIMESTAMPTZ

-- Indexes:
-- GIN on regions, sectors (fast overlap queries)
-- FTS on name, description, funder_name
-- B-tree on type, is_active, funder_id

3. Search API — POST /api/search

File: app/api/search/route.ts

Request (OrgInfo)

{
  website: string,         // org website URL
  linkedin?: string,       // org LinkedIn URL
  fundingPurpose: string,  // what funding is for
  fundingAmount: number,   // NZD amount sought
  market: string,          // 'nz' | 'au'
  regions: string[],       // region IDs the org operates in
  sectors: string[],       // sector IDs (e.g. ['health', 'youth'])
  orgType: string,         // 'registered-charity' | 'charitable-trust' | ...
  previousFunders?: string // free text
}

Pipeline

Step 1: DB Query — searchGrants(sectors, regions) queries the grants table with:

  • Region filter: empty regions = nationwide (no filter); non-empty = overlap match (&& operator)
  • Grants with NULL/empty regions always included (nationwide grants)
  • Order: sector overlap count DESC, funder name ASC
  • Returns all active grants matching (typically 500–4,500)

Step 2: Pass 1 — Triage (GPT-4o-mini)

  • All grants split into batches of 100
  • Each batch sent to GPT-4o-mini with the RELEVANCE_TRIAGE_PROMPT
  • Model classifies each grant as RELEVANT or SKIP
  • SKIP only when ≥90% confident: wrong sector, wrong region, wrong org type, dramatic amount mismatch, or non-grant content
  • On error, defaults all to RELEVANT (never drops grants due to API failure)
  • Concurrency: 15 parallel batches
  • Typically filters to 200–400 relevant grants

Step 3: Pass 2 — Scoring (GPT-4o-mini)

  • Relevant grants split into batches of 25
  • Each batch sent with SCORING_SYSTEM_PROMPT + org context
  • Model scores on 3 dimensions: alignment (0–10), ease (0–10), attainability (0–10)
  • Overall = (alignment × 0.5) + (attainability × 0.3) + (ease × 0.2)
  • First batch also generates orgSummary (2–3 sentence org description)
  • Concurrency: 15 parallel batches
  • Includes multi-check logic: URL quality, geographic eligibility, form-of-support match, regional relevance, deadline validation

Step 4: Filter & Return

  • Remove grants with alignment < 5.0
  • Assign deterministic IDs: g_ + sha256(funder|name|url)[:16]
  • Return SearchResult with grants array, orgSummary, timestamp, market, original inputs

Response (SearchResult)

{
  grants: GrantOpportunity[],  // scored grants
  orgSummary: string,          // AI-generated org description
  searchedAt: string,          // ISO timestamp
  market: string,              // 'nz'
  inputs?: OrgInfo             // echoed for re-use
}

4. Deep Search API — POST /api/deep-search

File: app/api/deep-search/route.ts

Triggered per-grant by the user. Performs live web research to find precise details the DB may lack.

Phase 1: Web Search (Serper)

  • 6–8 parallel search queries targeting: application forms, guidelines, deadlines, eligibility, past recipients
  • Site-specific searches on the funder's own domain
  • Collects unique URLs, prioritises funder's own domain

Phase 2: Content Extraction (Tavily)

  • Up to 10 discovered URLs extracted via Tavily
  • Full page content (bypasses JS-rendered pages, 403s)
  • Funder's own domain URLs prioritised

Phase 3: AI Analysis (GPT-4o)

  • All extracted content sent to GPT-4o (full model, not mini) for structured analysis
  • Extracts: precise amounts, open/close dates, eligibility criteria, application form URL & type, checklist of required/optional documents, key contacts, past recipient insights
  • Recalibrates all three scores (alignment, ease, attainability) with reasoning for each change

Phase 4: DB Write-back

  • Discovered fields written back to the grants table via writeDeepSearchUpdates()
  • Only fills NULL fields — existing data is never overwritten
  • Fields: application_form_url, amount_min, amount_max, deadline
  • Benefits all future searches (not just the user who triggered deep search)

Response (DeepSearchResult)

{
  grantId, grantName, funder, grantUrl, searchedAt,
  amountMin?, amountMax?, amountNotes?,
  applicationOpenDate?, applicationCloseDate?, dateNotes?,
  checklist: DeepSearchChecklistItem[],
  applicationFormUrl?, applicationFormType?, applicationFormNotes?,
  eligibilityCriteria: string[],
  scores: { alignment, ease, attainability, overall },
  scoreChanges: { alignment: { old, new, reason }, ease: ..., attainability: ... },
  additionalInfo?, keyContacts?, pastRecipientNotes?,
  sourcesUsed: { url, title }[]
}

5. Storage APIs

All storage uses Neon Postgres (no localStorage). Tables created on first use.

Endpoint Methods Description
/api/saved-searches GET, POST, DELETE Save/list/delete search results. Auto-saved after each search. Stores full SearchResult as JSONB.
/api/shortlist GET, POST, DELETE Add/remove/list shortlisted grants. Batch check endpoint. Grouped by search title.
/api/deep-search-results GET, POST, DELETE Store/retrieve deep search results. Batch lookup by grant IDs.
/api/applications GET, POST, PUT, DELETE Full CRUD for application tracking. Status transitions, notes, financial amounts.

8. Data Pipeline Scripts

All scripts run locally via DATABASE_URL="..." npx tsx scripts/<script>.ts. Some also require OPENAI_API_KEY and/or TAVILY_API_KEY.

Core Pipeline (run in order)

# Script What it does APIs used
1 import-charities.ts Fetches charities from NZ Charities Register API (grant-makers + name-matched trusts/foundations). Batched inserts (500/batch). Filters noise (PTAs, churches, sports clubs). Safe to re-run (ON CONFLICT). Register API
2 import-curated-funders.ts Imports 272 hand-curated funders from lib/markets/nz.ts. URL normalisation + dedup. Single-program funders merged with register entries; multi-program funders inserted separately. none
3 enrich-charities.ts For each unenriched funder: fetches curated_grant_url (or homepage), extracts grant programs via GPT-4o-mini, inserts into grants table. Only processes enriched_at IS NULL records. GPT-4o-mini
4 classify-grant-makers.ts GPT-4o classifies which charities are actual grant-makers vs grant-recipients incorrectly included. 1,762 non-grant-maker grants deactivated (is_active = false). GPT-4o
5 fill-missing-fields.ts Targeted re-enrichment pass for grants with missing amounts, deadlines, or form URLs. Fetches source pages and uses GPT-4o-mini to extract specific fields. GPT-4o-mini

Quality & Enhancement Scripts

Script What it does APIs used
retag-new-sectors.ts Re-tags all active grants with new sector IDs (economic-development, animal-welfare) using GPT-4o-mini. Only adds tags, never removes. Batches of 50. GPT-4o-mini
db-quality-fixes.ts Comprehensive fixes: merge errors (Fillmor House, Aoraki, etc.), homepage → grants page URL corrections, funder deduplication (NZCT, Four Winds, etc.), contamination grant deactivation, enriched_at resets. none (SQL only)
fill-apply-urls.ts Finds application form URLs for grants missing them. Part B: Tavily extract on source_url + GPT for URL extraction. Part A: Tavily search fallback (portal domains only). Dry-run by default (--write to persist). Tavily + GPT-4o-mini
fill-missing-fields-tavily.ts Re-enrichment using Tavily (bypasses 403s / JS-rendered pages). Groups by source_url for efficiency. Targets eligibility, amounts, deadlines. Only fills NULL fields. Tavily + GPT-4o-mini
seed-oct-grants.ts Manually seeds Otago Community Trust grants (uses Fluxx portal that can't be scraped). Hardcoded grant programs inserted directly. none

9. Scoring System Details

Pre-checks (applied before scoring)

  1. URL Quality Check: If content is a news article, press release, or past-recipients list rather than a grant program page → all scores zeroed.
  2. Geographic Eligibility: If grant is restricted to a country/region that isn't NZ → alignment=0, attainability=0, overall=0.

Scoring Dimensions

Dimension Weight Factors
Alignment 50% Grant purpose vs org mission, form of support match (cash vs in-kind), sector overlap, specific funding request match
Attainability 30% Competition level, eligibility fit, regional relevance, org type match, amount range match
Ease 20% Application complexity: simple online form (9–10) vs multi-stage with site visits (1–2)

Formula: Overall = (alignment × 0.5) + (attainability × 0.3) + (ease × 0.2), rounded to 1dp

Filter: Only grants with alignment ≥ 5.0 are returned to the user.

Score Recalibration (Deep Search)

When deep search runs, GPT-4o recalibrates all three scores based on the detailed information discovered. Each change includes a reason explaining why the score moved (e.g. "Application requires detailed financials and site visit — reduced ease from 7 to 4").
